Orcho's GitHub App automatically builds and maintains dependency graphs for your repositories, updating them on every push. Get comprehensive blast radius analysis and risk assessment for code changes with full visibility into how changes propagate through your codebase.

## How It Works

<Steps>

<Step title="Install Orcho GitHub App">
Install the app and grant repository access
</Step>

<Step title="Initial graph generation">
Orcho analyzes your codebase and builds a complete dependency graph:
- Import statements
- Function calls
- Class inheritance
- Module dependencies
- Configuration files
</Step>

<Step title="Developer pushes code">
Developer commits and pushes changes to GitHub
</Step>

<Step title="Webhook triggered">
GitHub webhook notifies Orcho of the push event
</Step>

<Step title="Graph updated">
Orcho analyzes the changed files and updates the dependency graph incrementally
</Step>

<Step title="Blast radius calculated">
For each changed file, Orcho calculates:
- Direct dependencies (files that import this file)
- Indirect dependencies (files that depend on direct dependencies)
- Total impact scope
</Step>

<Step title="Risk assessment">
Combined with other risk factors to produce overall risk score
</Step>

</Steps>

<Steps>

<Step title="Install the GitHub App">
Visit the [GitHub Marketplace](https://github.com/marketplace) and search for "Orcho Risk Assessment"

Or install directly: [Install Orcho](https://github.com/apps/orcho-risk)

<Check>
Works with GitHub.com and GitHub Enterprise Server
</Check>
</Step>

<Step title="Select repositories">
Choose which repositories to enable:
- All repositories (current and future)
- Only select repositories

<Warning>
Orcho needs read access to code and write access to checks/statuses
</Warning>
</Step>

<Step title="Contact for API key">
Email [support@orcho.ai](mailto:support@orcho.ai) to request an API key for your organization
</Step>

<Step title="Configure the app">
In GitHub, go to:
1. Settings → Integrations → Orcho Risk Assessment
2. Click "Configure"
3. Enter your API key
4. Configure risk thresholds
5. Enable PR checks
6. Save configuration

Orcho will automatically:
1. Analyze all enabled repositories
2. Generate dependency graphs (5-30 minutes depending on size)
3. Send you a notification when complete
</Step>

</Steps>

## Dependency Graph

### What Gets Analyzed

Orcho builds dependency graphs by analyzing:

<Tabs>

<Tab title="Python">
**Imports:**
```python
import requests
from datetime import datetime
from .utils import helper_function
```

**Function calls:**
```python
def main():
    result = helper_function()  # Dependency detected
    return process_result(result)
```

**Class inheritance:**
```python
class CustomHandler(BaseHandler):  # Dependency on BaseHandler
    pass
```
</Tab>

<Tab title="JavaScript/TypeScript">
**Imports:**
```javascript
import { useState } from 'react';
import { apiCall } from './api';
import utils from '../utils/helpers';
```

**Dynamic imports:**
```javascript
const module = await import('./dynamic-module');
```

**Requires:**
```javascript
const express = require('express');
```
</Tab>

<Tab title="Java">
**Imports:**
```java
import com.company.project.UserService;
import java.util.List;
```

**Class dependencies:**
```java
public class UserController {
    private UserService userService;  // Dependency
}
```
</Tab>

<Tab title="Go">
**Imports:**
```go
import (
    "fmt"
    "github.com/company/project/utils"
)
```

**Package dependencies:**
Automatically detected from go.mod
</Tab>

</Tabs>

### Graph Structure

The dependency graph is stored as a directed acyclic graph (DAG):

```json
{
  "repository": "company/api-server",
  "last_updated": "2026-01-08T14:30:00Z",
  "nodes": [
    {
      "file": "src/api/users.py",
      "type": "module",
      "imports": [
        "src/database/models.py",
        "src/auth/permissions.py"
      ],
      "imported_by": [
        "src/api/admin.py",
        "src/api/public.py"
      ]
    }
  ],
  "edges": [
    {
      "from": "src/api/users.py",
      "to": "src/database/models.py",
      "type": "import"
    }
  ],
  "stats": {
    "total_files": 347,
    "total_dependencies": 1205,
    "avg_dependencies_per_file": 3.47,
    "max_dependency_chain": 12
  }
}
```

## Pull Request Checks

### Automatic Risk Assessment

When a PR is opened or updated, Orcho automatically:

<Steps>

<Step title="Detect changed files">
Identifies all files modified in the PR
</Step>

<Step title="Calculate blast radius">
Using the dependency graph, finds all files that depend on changed files:

```
Changed files:
• src/auth/login.py

Direct dependencies (4 files):
• src/api/auth_endpoints.py
• src/middleware/auth_middleware.py  
• src/cli/login_command.py
• tests/test_auth.py

Indirect dependencies (12 files):
• src/api/users.py
• src/api/admin.py
• ... (10 more)

Total blast radius: 16 files
```
</Step>

<Step title="Assess risk">
Combines blast radius with other risk factors:
- Data sensitivity in changed code
- Clarity of PR description
- Type of changes (additions vs deletions)
</Step>

<Step title="Post check result">
Creates a GitHub Check with risk assessment:

<ResponseExample>
```
✓ Orcho Risk Assessment — PASSED

Risk Score: 0.42 (MEDIUM)

Risk Factors:
• Data Sensitivity: 0.35 (LOW)
• Input Clarity: 0.48 (MEDIUM)  
• Blast Radius: 0.45 (MEDIUM)

Blast Radius Details:
• Changed files: 2
• Direct dependencies: 4
• Indirect dependencies: 12
• Total impacted files: 16

Recommendations:
• REVIEW_RECOMMENDED
• Add tests for impacted files
• Update documentation

View detailed report →
```
</ResponseExample>
</Step>

</Steps>

### Check Configuration

Configure when checks run and what they do:

<ParamField path="run_on_pr" type="boolean" default="true">
Run risk assessment on pull requests

Default: `true`
</ParamField>

<ParamField path="run_on_push" type="boolean" default="false">
Run risk assessment on direct pushes to protected branches

Default: `false`
</ParamField>

<ParamField path="block_high_risk" type="boolean" default="false">
Block merging of high-risk PRs (&ge; 0.6)

Default: `false`
</ParamField>

<ParamField path="require_approval_threshold" type="number" default="0.7">
Risk score that requires additional approval

Default: `0.7`
</ParamField>

<ParamField path="comment_on_pr" type="boolean" default="true">
Post a comment with risk details

Default: `true`
</ParamField>

### Status Checks

Orcho creates GitHub Status Checks that can be required for merging:

**Low Risk (&lt; 0.4):**
```
✓ orcho/risk-assessment — Low risk detected
```

**Medium Risk (0.4-0.59):**
```
✓ orcho/risk-assessment — Medium risk, review recommended
```

**High Risk (0.6-0.79):**
```
⚠ orcho/risk-assessment — High risk, approval required
```

**Critical Risk (&ge; 0.8):**
```
✗ orcho/risk-assessment — Critical risk, changes required
```

## Branch Protection Rules

### Require Orcho Checks

Make Orcho risk assessment required before merging:

<Steps>

<Step title="Go to branch settings">
1. Repository → Settings → Branches
2. Find your protected branch (e.g., `main`)
3. Click "Edit" on branch protection rule
</Step>

<Step title="Require status checks">
1. Check "Require status checks to pass before merging"
2. Search for "orcho/risk-assessment"
3. Add the check
4. Save changes
</Step>

<Step title="Configure thresholds">
In Orcho settings, set what constitutes a passing check:

```json
{
  "pass_threshold": 0.6,  // PRs with score < 0.6 pass
  "fail_threshold": 0.8,  // PRs with score >= 0.8 fail
  "warn_threshold": 0.6   // PRs with 0.6-0.79 warn but allow
}
```
</Step>

</Steps>

### Custom Rules

Create custom rules based on blast radius:

**Block if too many files affected:**
```json
{
  "rule": "max_blast_radius",
  "max_affected_files": 50,
  "action": "block"
}
```

**Require review if critical files affected:**
```json
{
  "rule": "critical_files",
  "files": [
    "src/auth/**",
    "src/database/migrations/**",
    "config/production.yml"
  ],
  "action": "require_review"
}
```

**Warning for large dependency chains:**
```json
{
  "rule": "dependency_depth",
  "max_depth": 8,
  "action": "warn"
}
```

## Graph Updates

### Incremental Updates

Orcho updates the dependency graph incrementally on each push:

<Steps>

<Step title="Detect changed files">
From git diff: `src/utils/helpers.py` modified
</Step>

<Step title="Analyze changes">
```python
# Added new import
import logging  # New dependency

# Removed old import  
# from datetime import datetime  # Dependency removed
```
</Step>

<Step title="Update graph">
- Add edge: `helpers.py → logging`
- Remove edge: `helpers.py → datetime`
- Update: Files that import `helpers.py` now transitively depend on `logging`
</Step>

<Step title="Recalculate metrics">
Update stats:
- Total dependencies: 1206 → 1206 (added 1, removed 1)
- Files depending on helpers.py: unchanged
</Step>

</Steps>

### Full Rebuild

Trigger a full graph rebuild:

1. Go to repository settings
2. Orcho → Advanced
3. Click "Rebuild Dependency Graph"
4. Wait 5-30 minutes (depending on repo size)

**When to rebuild:**
- After major refactoring
- If graph seems out of sync
- After changing large number of files
- Monthly for active repos (recommended)

## API Access

### GraphQL API

Query your dependency graph via API:

```graphql
query GetDependencyGraph($repo: String!) {
  repository(fullName: $repo) {
    dependencyGraph {
      nodes {
        file
        imports
        importedBy
      }
      stats {
        totalFiles
        totalDependencies
      }
    }
  }
}
```

### REST API

Get blast radius for specific files:

```bash
curl -X POST https://api.orcho.ai/api/v1/blast-radius \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "repo_full_name": "company/api-server",
    "files": ["src/auth/login.py"]
  }'
```

**Response:**
```json
{
  "changed_files": ["src/auth/login.py"],
  "direct_dependencies": 4,
  "indirect_dependencies": 12,
  "total_impact": 16,
  "affected_files": [
    "src/api/auth_endpoints.py",
    "src/middleware/auth_middleware.py",
    "..."
  ]
}
```

## Best Practices

<Steps>

<Step title="Start with smaller repos">
Install on 1-2 smaller repositories first to validate before org-wide rollout
</Step>

<Step title="Set appropriate thresholds">
Tune risk thresholds based on your team's tolerance:
- Strict: Block at 0.5
- Balanced: Block at 0.7
- Permissive: Block at 0.8+
</Step>

<Step title="Review blast radius regularly">
Use blast radius insights to:
- Identify tightly coupled code
- Plan refactoring efforts
- Improve code architecture
</Step>

<Step title="Integrate with CI/CD">
Use Orcho risk scores in your deployment pipeline:
```yaml
# .github/workflows/deploy.yml
- name: Check Risk Score
  run: |
    if [ $ORCHO_RISK_SCORE -ge 0.7 ]; then
      echo "Risk too high for auto-deploy"
      exit 1
    fi
```
</Step>

<Step title="Keep graphs updated">
Set up weekly full rebuilds for active repositories
</Step>

</Steps>
